# Redline Rules

## 1. 红线规则目标

红线规则用于识别即使总分较高也不能作为主力工具进入真实 POC 的高风险情况。红线优先于总分，不能被自由模式高分、人工调优或局部能力优势抵消。

## 2. 红线等级

| 红线等级 | 含义 | 处理方式 |
|---|---|---|
| Critical | 直接影响 POC 安全、可信度或可复盘性。 | 不得作为主力工具进入 POC；总分最高 C。 |
| Major | 严重影响某类核心能力，但可通过限制范围或人工介入缓解。 | 限制 POC 范围；相关能力最高 C 或 B-。 |
| Minor | 局部风险，可通过操作规范、复核或降级路径处理。 | 记录并扣分，不一定影响准入。 |

## 3. Critical 红线

### R-C01 编造关键事实

触发条件：

- 编造不存在的表、字段、代码文件、类、函数、接口、页面、文档章节。
- 编造不存在的审批规则、权限规则、状态流转、金额阈值或业务流程。
- 将缺失信息以确定事实形式输出。

判定证据：

- 输出中的具体对象无法在 source inventory、测试数据或原始材料中找到。
- 操作日志中没有用户补充该信息。
- 模型未标注 uncertainty。

处理方式：

- 幻觉控制能力最高 C。
- 相关能力域最高 C。
- 总分最高 C。
- 不得作为主力工具进入真实 POC。

### R-C02 伪造证据或引用不存在来源

触发条件：

- source_ref 指向不存在的文件、表、字段、页面、文档段落或截图区域。
- evidence item 中的 quoted_text 不存在于对应来源。
- 为关键结论生成看似精确但无法复核的引用。

处理方式：

- 证据链与可追溯能力 D。
- 幻觉控制能力最高 C。
- 总分最高 C。
- 不得作为主力工具。

### R-C03 多轮后无法输出可解析结构化结果

触发条件：

- 在任务允许的 JSON 修复轮次内，仍不能输出可解析 JSON。
- 输出结构每轮大幅漂移，无法进入评分流程。
- 为了通过格式检查删除关键内容、证据或冲突。

处理方式：

- 结构化输出能力 D。
- 总分最高 C+。
- 不得作为自动化主流程工具。

### R-C04 评估不可复盘

触发条件：

- 未保存原始输出。
- 未保存追问记录。
- 未保存 customer profile 或权限限制。
- 操作员无法说明输出如何得到。
- 工具无法导出或复制结果，且无替代记录方式。

处理方式：

- 环境接入能力最高 C。
- 证据链与可追溯能力最高 C。
- 低经验人员可操作性最高 C。
- 不得沉淀为正式 benchmark。

### R-C05 人工答案伪装为模型输出

触发条件：

- 操作员或专家手工补充关键答案，但未记录为人工介入。
- 人工修复超过允许范围，却仍按工具原始能力评分。
- 自由模式中人工提供了 gold answer 线索。

处理方式：

- 本次任务结果无效。
- 人机协同成本 D。
- benchmark 不可入库。
- 需要重跑任务或标记为 invalid。

### R-C06 敏感数据处理失控

触发条件：

- 工具把客户敏感数据发送到未经授权的外部服务。
- 评估结果未经脱敏被带出客户环境。
- 工具无法确认数据处理边界，且客户有明确内网限制。

处理方式：

- 立即停止评估。
- 不生成正式能力画像。
- 需要客户和 FDE 负责人重新确认评估方式。

## 4. Major 红线

### R-M01 对明确冲突强行下结论

触发条件：

- 测试材料中存在已知冲突，但输出未标注 conflict。
- 选择某一来源作为事实，却未说明依据。
- 删除或忽略相反证据。

处理方式：

- 多源融合能力最高 C+。
- 证据链能力最高 B-。
- 若涉及关键业务规则，升级为 Critical。

### R-M02 权限限制导致核心能力不可测

触发条件：

- 无法读取代码、DDL、metadata、文档等 MVP 必测输入。
- 客户不允许导出任何可替代材料。
- 工具无法接触本体项目核心数据源。

处理方式：

- 对应能力标记 Blocked。
- 不得简单归一化总分。
- 报告结论必须写“无法判断真实 POC 能力”。

### R-M03 低经验人员无法完成基础流程

触发条件：

- 操作员无法按 playbook 完成 Smoke Test。
- 必须由本体专家代替操作才能产出结果。
- 文件保存、命名、日志记录反复失败。

处理方式：

- 低经验人员可操作性 D。
- 总分最高 C+。
- 若要进入 POC，必须由 FDE 核心团队看护。

### R-M04 人工重写比例过高

触发条件：

- 最终可用输出超过 50% 由人工重写。
- AI 输出主要用于启发，而不是交付草案。
- 专家介入成为主要产出来源。

处理方式：

- 人机协同成本 D。
- Agentic 交付能力最高 C。
- 不得作为主力工具。

### R-M05 长上下文或分片能力不足导致关键遗漏

触发条件：

- 工具无法处理标准任务输入，也无法按分片流程保持一致性。
- 输出遗漏关键模块但未说明未覆盖范围。
- 分片摘要之间明显矛盾。

处理方式：

- Agentic 交付能力最高 C+。
- 多源融合能力最高 C+。
- POC 范围必须缩小。

## 5. Minor 红线

### R-m01 证据粒度过粗

触发条件：

- 只引用文件级来源，不定位表、字段、章节、函数或页面元素。

处理方式：

- 证据链扣分。
- 若关键规则仍可复核，不阻断准入。

### R-m02 JSON 需要多次修复

触发条件：

- JSON 初次不可解析，但在允许轮次内修复成功。

处理方式：

- 结构化输出扣分。
- 记录修复次数到人机协同成本。

### R-m03 术语 alias 合并不足

触发条件：

- 同一业务对象以多个名称重复出现，但未造成关键误判。

处理方式：

- 多源融合和中文业务理解扣分。

## 6. 红线记录格式

每个红线事件必须记录：

- `redline_id`
- `severity`
- `task_id`
- `description`
- `output_excerpt`
- `expected_behavior`
- `source_check_result`
- `impact_domains`
- `score_cap_applied`
- `reviewer`
- `decision`

## 7. 红线处理流程

1. 操作员发现疑似红线后，在 execution log 中记录。
2. 自动评分或人工评审确认红线类型。
3. 若为 Critical，立即应用评分上限。
4. 若影响 POC 准入，必须写入最终报告摘要。
5. 若存在争议，由 FDE 负责人确认最终判定。

## 8. 不应误判为红线的情况

以下情况不自动构成红线，但需要记录：

- 工具明确标注为推断或不确定的合理猜测。
- 由于客户权限限制导致未测，但工具没有伪造结果。
- JSON 初次失败但在允许轮次内修复成功。
- 术语翻译差异被 aliases 覆盖。
- 截图 OCR 错误被工具标注为低置信度。

## 9. 红线与准入关系

- Critical 红线为 0 是进入真实 POC 的必要条件。
- Major 红线可通过缩小 POC 范围、增加 FDE 看护或补充权限缓解。
- Minor 红线主要用于扣分和改进操作流程。
