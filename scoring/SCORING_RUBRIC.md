# Scoring Rubric

## 1. 评分目标

本评分体系用于判断客户现场 AI 工具组合是否足以支撑本体项目交付。评分不是模型排行榜，而是服务于以下决策：

- 是否允许进入真实客户 POC。
- 适合承担哪些本体建设任务。
- 哪些能力必须由 FDE 核心团队介入。
- 客户需要补充哪些权限、数据源或工具能力。
- 该工具组合能否沉淀为可复用能力画像。

## 2. 总分结构

总分为 100 分，覆盖 15 个能力域。

| 能力域 | 权重 | MVP 是否必测 | 说明 |
|---|---:|---|---|
| 环境接入能力 | 6 | 是 | 判断是否能读取、保存、记录和降级。 |
| 代码理解能力 | 8 | 是 | 判断能否从代码抽取业务对象和规则。 |
| 数据库 Metadata 理解能力 | 8 | 是 | 判断能否从 DDL/metadata 抽取实体、字段和关系。 |
| 数据样例分析能力 | 6 | 是 | 判断能否从样例数据识别状态、枚举和异常。 |
| UI / 截图理解能力 | 5 | 否 | V1 必测；MVP 标注 N/A 或 Blocked。 |
| Web 递归分析能力 | 5 | 否 | V1 必测；MVP 标注 N/A 或 Blocked。 |
| 文档理解能力 | 7 | 是 | 判断能否从中文业务文档抽取概念和规则。 |
| 多源融合能力 | 10 | 是 | 判断能否合并多源证据并处理冲突。 |
| 结构化输出能力 | 9 | 是 | 判断能否稳定输出可解析 JSON。 |
| Agentic 交付能力 | 8 | 部分 | MVP 测基础分阶段能力，V2 深测。 |
| 证据链与可追溯能力 | 8 | 是 | 判断本体元素是否能追溯到来源。 |
| 低经验人员可操作性 | 5 | 是 | 判断初级交付人员能否按手册执行。 |
| 人机协同成本 | 4 | 是 | 判断 AI 是否真正降低交付成本。 |
| 幻觉控制能力 | 6 | 是 | 判断是否编造事实、规则和来源。 |
| 中文政企业务理解能力 | 5 | 是 | 判断中文政企业务语义理解质量。 |
| 合计 | 100 |  |  |

## 3. 等级标准

| 等级 | 分数 | 含义 | 项目策略 |
|---|---:|---|---|
| A+ | 90-100 | 可作为主力工具，表现接近理想交付助手。 | 可进入 POC 主流程，FDE 专家抽查。 |
| A- | 85-89 | 可作为主力工具，但个别能力需校验。 | 可进入 POC 主流程，关键节点复核。 |
| B+ | 80-84 | 能承担核心工作，人工校验成本可接受。 | 可作为辅助主力。 |
| B | 75-79 | 基本可用于 POC，需 FDE 看护。 | 可承担单源分析和多源初稿。 |
| B- | 70-74 | 勉强准入，限制条件必须明确。 | 可进入有限范围 POC。 |
| C+ | 65-69 | 局部可用，但不适合主流程。 | 只能承担低风险辅助任务。 |
| C | 60-64 | 需要专家强介入。 | 不建议进入标准 POC 主流程。 |
| C- | 55-59 | 产出价值有限。 | 仅可做探索或演示。 |
| D | 0-54 | 不适合本体项目交付。 | 不建议使用，需更换工具或权限。 |

## 4. 自动评分范围

自动评分应优先覆盖可客观判断的项目。

### 4.1 Schema 合规性

- JSON 是否可解析。
- 是否符合 ontology/evidence/task/scoring 结构要求。
- 必填字段是否完整。
- 字段类型是否正确。
- 枚举值是否落在允许范围。

### 4.2 Gold 对比

- 核心概念召回。
- 关键属性召回。
- 关系识别。
- 事件识别。
- 规则识别。
- 状态识别。
- 角色/权限识别。

评分必须支持：

- acceptable aliases。
- 中文同义词。
- 字段名、业务名、英文名映射。
- partial credit。
- 合理抽象差异。

### 4.3 Evidence 检查

- 每个核心元素是否有 source_ref。
- source_ref 是否指向存在的文件、表、字段、页面或文档段落。
- evidence 类型是否与来源匹配。
- 关键规则是否有直接证据。
- 是否存在伪造来源。

### 4.4 重复和冲突检查

- 重复概念数量。
- alias 未合并数量。
- 已知冲突召回率。
- 冲突是否有来源说明。
- 是否强行消除冲突。

### 4.5 幻觉检查

- 编造文件、表、字段、接口、规则、页面、角色。
- 引用不存在来源。
- 对缺失信息给出确定答案。
- 将样例观察上升为全局规则。

## 5. 人工评分范围

人工评分覆盖自动评分无法可靠判断的质量。

| 人工评分项 | 评分重点 |
|---|---|
| 业务合理性 | 输出是否符合采购、审批、合同、履约等企业业务常识。 |
| 概念抽象质量 | 是否把表、代码、文档提升为合适粒度的业务概念。 |
| 多源融合质量 | 是否能去重、合并 alias、保留冲突和不确定性。 |
| 冲突处理能力 | 是否识别冲突、解释来源、给出合理处理建议。 |
| 不确定性表达 | 是否区分事实、推断和未知。 |
| 交付可用性 | 输出是否能进入本体草案评审或客户讨论。 |
| 低经验人员可复现性 | 操作员是否按手册可重复产出类似结果。 |
| 中文业务理解 | 是否准确理解中文政企业务术语和流程。 |

人工评分建议使用 1-5 分锚点：

| 分数 | 含义 |
|---:|---|
| 5 | 可直接用于交付审核，仅需少量润色。 |
| 4 | 基本可用，有少量明显修正点。 |
| 3 | 局部可用，需要专家较多修正。 |
| 2 | 产出质量弱，只能作为参考。 |
| 1 | 不可用或误导性强。 |

## 6. 能力域评分建议

每个能力域建议按以下比例综合：

- 自动评分：60%。
- 人工评分：40%。

例外：

- 人机协同成本：自动记录 70%，人工判断 30%。
- 低经验人员可操作性：过程记录 50%，人工反馈 50%。
- 中文政企业务理解：自动术语召回 40%，人工业务判断 60%。
- 多源融合能力：自动匹配 50%，人工融合质量 50%。

## 7. 评分上限规则

评分上限用于防止受限环境下的总分被高估。

| 限制或失败 | 上限规则 |
|---|---|
| 工具不能读取文件，只能手工复制 | 总分最高 B-；多源融合最高 C+。 |
| 工具不能读取目录 | 代码理解最高 B-；Agentic 最高 B-。 |
| 工具不能执行命令 | Agentic 交付最高 B；环境接入最高 B。 |
| 工具不能保存中间结果 | 证据链最高 B-；Agentic 最高 B-。 |
| 工具不能稳定输出 JSON | 结构化输出最高 C；总分最高 C+。 |
| 允许修复轮次内仍不可解析 JSON | 结构化输出 D；不得作为主力工具。 |
| 无 evidence 或 source_ref | 证据链最高 C；总分最高 B-。 |
| source_ref 大量不存在 | 触发红线；总分最高 C。 |
| 出现严重幻觉 | 触发红线；总分最高 C，且不得作为主力工具。 |
| 工具不能看图 | UI 能力 N/A 或最高 C；若 POC 强依赖 UI，则不得无条件准入。 |
| 工具不能访问 Web | Web 能力 N/A 或最高 B-；若 POC 强依赖页面分析，则需降级范围。 |
| 工具不能连接数据库 | DB 能力最高 B，前提是可提供 DDL/metadata 导出。 |
| 无长上下文且不能分片 | Deep Agentic 最高 C+。 |
| 低经验操作员无法完成 Smoke Test | 低经验可操作性 D；总分最高 C+。 |
| 人工重写比例超过 50% | 人机协同成本 D；总分最高 C+。 |

## 8. 红线处理

红线规则详见 `REDLINE_RULES.md`。评分时必须遵守：

- 红线优先于总分。
- 红线必须记录证据。
- 红线不得被自由模式高分抵消。
- 红线失败后，报告中仍可保留局部能力评分，但准入结论必须受限。

## 9. N/A 与 Blocked 处理

| 标记 | 使用场景 | 对总分影响 |
|---|---|---|
| N/A | 当前测试范围没有覆盖该能力，例如 MVP 暂未测截图。 | 可计算已测能力归一化分，但报告必须注明未测风险。 |
| Blocked | 原本应测，但因客户权限或工具限制无法执行。 | 不应简单归一化；必须触发评分上限或准入条件。 |

## 10. 固定模式与自由模式评分

每个任务建议记录两套分数：

- `fixed_mode_score`：使用统一 prompt，适合横向比较。
- `free_mode_score`：允许合理适配，评估交付上限。

报告中必须同时展示：

- 固定模式基础能力。
- 自由模式交付上限。
- 两者差距。
- 自由模式增加的人机协同成本。

自由模式不能覆盖固定模式红线，不能作为唯一横评依据。

## 11. 总分计算流程

1. 检查任务是否完成。
2. 检查红线失败项。
3. 执行 schema / source_ref / gold / hallucination 自动评分。
4. 填写人工评分表。
5. 按能力域计算 raw_score。
6. 应用评分上限规则。
7. 映射 A/B/C/D 等级。
8. 汇总能力矩阵。
9. 根据 `ENTRY_CRITERIA.md` 给出准入建议。

## 12. 报告必须展示的评分信息

- 总分和总等级。
- 每个能力域 raw score 和 grade。
- 自动评分与人工评分拆分。
- N/A 和 Blocked 项。
- 红线失败项。
- 评分上限原因。
- 人机协同成本。
- 推荐使用场景。
- 不推荐使用场景。
- 是否适合进入真实 POC。
