# Implementation Plan

## 1. 项目目标

构建一套可长期复用的“本体项目 AI 工具能力评估工具包”，用于判断客户内网提供的 AI 工具组合是否足以支撑 FDE 团队的本体项目交付。

核心判断问题：

> 在客户真实内网、权限限制、数据源限制、工具能力限制和操作员水平限制下，这套 AI 工具组合能否稳定产出可用于本体项目交付的结构化结果？

## 2. 总体定位

### 它是

- 本体项目 AI 工具能力评估工具包。
- 客户现场 AI 能力准入评估工具。
- FDE 团队判断客户内网 AI 工具能否支撑本体交付的标准方法。
- 可复用、可配置、可离线执行的测试框架。

### 它不是

- 单一客户 POC 测试包。
- 单纯模型 benchmark。
- 单纯 AI 编码工具测评。
- 学术型本体评测集。
- 聊天能力或代码补全能力评测。

## 3. 被测对象定义

被测对象是完整组合：

```text
客户环境 + AI 工具 + 底层模型 + 权限边界 + 可用数据源 + 操作人员 + 标准任务包
```

每次结果都应形成一个工具组合能力画像，不应脱离权限、环境和操作方式单独评价模型。

## 5. 核心包与适配包

- `core/`：能力模型、任务协议、评分规则、红线规则、方法论。
- `datasets/`：通用数据集和行业扩展数据集。
- `adapters/`：代码、数据库、文档、截图、Web、多源融合等任务适配说明。
- `customer_profiles/`：客户现场工具、模型、权限、网络、数据源配置。
- `results/`：每次评估的原始输出、标准化结果、评分和报告。
- `benchmark_history/`：长期沉淀的工具组合能力画像。

## 6. 标准能力模型

MVP 起步即保留完整 15 个能力域：

1. 环境接入能力
2. 代码理解能力
3. 数据库 Metadata 理解能力
4. 数据样例分析能力
5. UI / 截图理解能力
6. Web 递归分析能力
7. 文档理解能力
8. 多源融合能力
9. 结构化输出能力
10. Agentic 交付能力
11. 证据链与可追溯能力
12. 低经验人员可操作性
13. 人机协同成本
14. 幻觉控制能力
15. 中文政企业务理解能力

MVP 可暂不实测 UI/Web，但必须保留能力域和评分上限规则，避免误判工具总能力。

## 7. 测试数据集设计

### MVP 数据集

通用业务域：企业采购审批与合同履约管理系统。

覆盖对象：采购申请、采购需求、供应商、合同、合同条款、订单、到货验收、发票、付款申请、审批流程、预算、组织部门、角色权限、履约事件、风险预警。

### 难度阶梯

- Level 1：字段清晰、结构简单、命名一致。
- Level 2：多模块、多表关联。
- Level 3：命名不一致、历史遗留、冗余字段。
- Level 4：多源冲突、缺文档、错误字段、截图与数据库不一致。

MVP 只创建 Level 1/2 设计，不在本阶段生成实际数据。

## 8. 测试任务设计

### Smoke Test

目标：1-2 小时判断基本可用性。

任务包括：

- ST-00 环境能力登记
- ST-01 DDL 到本体草案
- ST-02 文档到规则和流程
- ST-03 多源小融合
- ST-04 幻觉红线测试

### Standard Test

目标：0.5-1 天判断是否可承担真实 POC 的部分工作。

任务包括：

- STD-01 代码模块理解
- STD-02 DB metadata 理解
- STD-03 样例数据语义分析
- STD-04 文档规则抽取
- STD-05 多源融合
- STD-06 冲突识别
- STD-07 Evidence map 生成
- STD-08 JSON 修复与稳定性
- STD-09 操作员复现测试

### Deep Agentic Test

V2 阶段再实现，用于判断是否接近 Claude Code / Codex 的自动化流水线能力。

## 9. 固定提示词模式与自由发挥模式

### 固定提示词模式

- 所有工具使用同一套 prompt。
- 用于横向比较基础能力。
- 限定追问次数。
- 输出必须符合指定结构。

### 自由发挥模式

- 允许操作员根据工具特点做合理适配。
- 用于评估真实交付上限。
- 必须记录所有人工介入、prompt 改写和重试过程。

最终报告应同时展示 fixed mode 与 free mode，不得混为一个分数。

## 10. 降级路径

工具可能受限时必须记录限制和评分上限：

- 不能执行命令：Agentic 能力最高 B。
- 不能读取目录：代码/多源能力最高 B-。
- 不能看图片：UI 能力 N/A 或最高 C。
- 不能访问 Web：Web 能力最高 B-。
- 不能连接数据库：DB 能力最高 B。
- 不能稳定输出 JSON：总分最高 C+。
- 没有长上下文：Deep Agentic 最高 B。
- 不能保存中间结果：证据链和 Agentic 能力最高 B-。

## 11. 输出结构设计

后续 schema 应至少支持：

- concept
- attribute
- relation
- event
- rule
- action
- state
- role
- permission
- evidence
- source_ref
- confidence
- uncertainty
- aliases
- conflict
- mapping
- provenance

Evidence map 应支持代码、DDL、metadata、样例数据、截图、文档、Web 页面等多源证据。

## 12. 评分体系

总分 100 分，自动评分与人工评分结合。

自动评分覆盖：schema 合规、核心概念召回、属性召回、关系识别、规则识别、证据引用、来源合法性、重复概念、幻觉字段。

人工评分覆盖：业务合理性、概念抽象质量、多源融合质量、冲突处理、不确定性表达、交付可用性、低经验人员可复现性。

等级：A、B、C、D，并支持 B- 作为客户现场准入关键线。

## 13. MVP / V1 / V2

### MVP

- 通用采购业务系统数据集设计。
- 代码、DDL、样例数据、文档四类输入。
- Smoke Test 和 Standard Test。
- 基础 schema 设计说明、评分规则、报告模板、操作手册。

### V1

- 增加截图、Web 页面快照。
- 增加 Level 3/4 混乱数据。
- 增加行业数据集。
- 增加工具画像和历史 benchmark。

### V2

- 增加 Deep Agentic Test。
- 增加变更影响测试。
- 增加跨客户对比 dashboard。
- 增加更完整的本体质量评估和操作员认证。
