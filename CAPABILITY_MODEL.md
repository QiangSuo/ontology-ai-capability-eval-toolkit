# Capability Model

## 1. 目标与适用范围

本能力模型用于评估客户现场 AI 工具组合是否能够支撑本体项目交付。评估对象不是单独模型，也不是单独 AI 编码工具，而是：

```text
客户环境 + AI 工具 + 底层模型 + 权限边界 + 可用数据源 + 操作人员 + 标准任务包
```

本文件定义 15 个能力域。每个能力域都必须能被测试任务验证、能被评分、能被记录限制，并能映射到真实 POC 准入判断。

## 2. 通用评分口径

每个能力域采用 0-100 分原始分，并映射到 A/B/C/D 等级。若能力因环境限制不可测，应使用 `N/A` 或 `Blocked`，不得用主观猜测补分。

| 等级 | 分数区间 | 交付含义 |
|---|---:|---|
| A | 85-100 | 可作为主力能力使用，低经验人员可按流程稳定产出，FDE 专家做抽查即可。 |
| B | 70-84 | 可作为重要辅助能力使用，核心任务可完成，但需要人工校验和局部修正。 |
| C | 55-69 | 只能局部使用，适合单源摘要、字段解释或辅助分析，不适合作为主流程。 |
| D | 0-54 | 不建议用于本体项目交付，输出不稳定、不可复盘或幻觉风险高。 |
| N/A | 不计分 | 当前测试范围未覆盖该能力，报告中必须说明未测原因。 |
| Blocked | 不计分但影响准入 | 因客户权限或工具限制无法测试关键能力，必须触发评分上限或准入限制。 |

### 2.1 能力域评分字段

每个能力域的评分记录必须至少包含：

- `domain_id`：能力域编号。
- `domain_name`：能力域名称。
- `raw_score`：0-100 原始分。
- `grade`：A/B/C/D/N/A/Blocked。
- `auto_score`：自动评分部分。
- `manual_score`：人工评分部分。
- `redline_failures`：红线失败项。
- `cap_reasons`：评分上限原因。
- `evidence_paths`：支持该评分的结果文件、日志或人工评分表。
- `notes`：评审备注。

### 2.2 自动评分与人工评分关系

- 自动评分用于检查结构化、召回、引用合法性、重复、明显幻觉。
- 人工评分用于判断业务合理性、抽象质量、冲突处理、交付可用性。
- 自动评分不能替代人工业务判断。
- 红线失败项优先于总分。

## 3. 能力域定义

## 3.1 环境接入能力

### 测试目标

判断客户工具组合是否能访问测试材料、读取文件、保存输出、记录日志，并支持后续本体评估流程。

### 输入材料

- customer profile 模板。
- 测试包目录清单。
- 文件读取测试材料。
- 权限 checklist。
- 操作员环境记录。

### 输出要求

- `environment_capability.json` 或等价记录。
- 明确记录：是否联网、是否可读文件、是否可递归读目录、是否可执行命令、是否可看图、是否可访问 Web、是否可连接数据库、是否可保存中间结果。
- 明确记录每个限制的降级路径。

### 自动评分项

- customer profile 完整率。
- 权限字段填写完整率。
- 文件读取成功率。
- 输出保存成功率。
- 日志留存完整率。

### 人工评分项

- 操作员是否能独立完成环境登记。
- 工具限制是否描述清楚。
- 降级路径是否可执行。
- 是否存在不可复盘风险。

### 红线失败项

- 无法访问任何输入材料，且无法通过手工复制替代。
- 无法保存原始输出和日志。
- 工具限制被隐瞒或未记录，导致评分不可解释。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 可读目录、读文件、保存输出，可执行必要命令或工具动作，日志完整。 |
| B | 可读文件并保存输出，但命令执行、数据库或浏览器能力受限。 |
| C | 主要依赖手工复制输入输出，仍可完成 Smoke Test。 |
| D | 输入输出不稳定，无法形成可复盘结果包。 |

## 3.2 代码理解能力

### 测试目标

判断工具能否从源代码中识别业务对象、接口、流程、状态、权限逻辑和规则候选，并将其映射为本体元素。

### 输入材料

- Controller / API route 文件。
- Service 文件。
- Entity / Model 文件。
- Repository / DAO 文件。
- Validation / Permission 相关代码。
- 代码目录清单或 manifest。

### 输出要求

- `code_ontology.json` 或等价结构化输出。
- 至少包含 concept、attribute、relation、event、rule、action、role、permission。
- 每个关键元素必须带 code evidence：文件路径、类/函数名、行号或符号名。
- 对无法确认的业务含义必须填 uncertainty。

### 自动评分项

- 核心业务类召回率。
- API / 方法识别率。
- 代码 source_ref 合法率。
- 规则候选识别率。
- 编造类、方法、文件数量。

### 人工评分项

- 是否能从技术结构抽象出业务概念。
- 是否识别跨文件调用和业务流程。
- 是否区分技术对象与业务对象。
- 是否能发现代码中的权限或校验规则。

### 红线失败项

- 编造不存在的文件、类、函数或接口。
- 把技术日志、DTO、工具类错误抽象为核心业务概念且未标注不确定。
- 无证据声称存在关键审批、权限或风控规则。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 能跨模块理解业务流程，准确抽取对象、接口、规则和证据。 |
| B | 能理解主要模块和核心对象，但跨模块关系需人工校验。 |
| C | 只能做文件级摘要，业务抽象弱，证据粒度较粗。 |
| D | 大量编造或误读代码结构，不适合用于本体抽取。 |

## 3.3 数据库 Metadata 理解能力

### 测试目标

判断工具能否从 DDL、metadata 和字段注释中识别实体、字段、主外键、状态、枚举和业务语义。

### 输入材料

- DDL 文件。
- metadata JSON 或数据字典。
- 字段注释。
- 表关系说明。
- 数据库导出 manifest。

### 输出要求

- `db_ontology.json`。
- 表到 concept 的映射。
- 字段到 attribute 的映射。
- 外键、弱外键或命名关系到 relation 的映射。
- 枚举字段、状态字段、金额字段、时间字段和组织字段标注。
- 每个映射必须有 database evidence。

### 自动评分项

- 关键表召回率。
- 关键字段召回率。
- 主外键识别率。
- 状态/枚举字段识别率。
- 表字段 source_ref 合法率。
- 编造表、字段、约束数量。

### 人工评分项

- 表到业务实体的抽象质量。
- 字段语义解释是否合理。
- 是否识别历史字段、冗余字段和技术字段。
- 是否避免机械表字段翻译。

### 红线失败项

- 编造不存在的表、字段、外键或约束。
- 将明显技术字段错误识别为核心业务属性且未标注不确定。
- 忽略 DDL 中明确存在的核心表。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 能准确识别实体、字段、关系、状态、枚举和业务含义。 |
| B | 表字段识别准确，关系和语义解释需要少量校验。 |
| C | 主要是机械转写表结构，业务抽象有限。 |
| D | 表结构误读严重或编造数据库对象。 |

## 3.4 数据样例分析能力

### 测试目标

判断工具能否从样例数据中识别状态值、枚举值、异常值、缺失值、数据分布和规则候选，并区分事实、推断和不确定性。

### 输入材料

- CSV/Excel 样例数据。
- 字段说明。
- 数据量说明。
- 脱敏说明。

### 输出要求

- `sample_insights.json`。
- 识别枚举、状态、异常值、空值、格式模式。
- 对规则候选标注 `evidence_strength` 或 `confidence`。
- 明确说明样例数据不能证明的内容。

### 自动评分项

- 已知枚举值召回率。
- 状态值识别率。
- 异常值识别率。
- 样例来源引用合法率。
- 无证据规则数量。

### 人工评分项

- 推断是否克制。
- 是否区分样例观察与业务规则。
- 是否发现有价值的数据质量问题。
- 是否能将样例值映射到本体属性或状态。

### 红线失败项

- 把少量样例直接当作全量业务规则。
- 无证据声称字段只有某些取值。
- 忽略明显异常值并给出错误结论。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 能识别数据模式、异常和规则候选，并清楚表达证据强弱。 |
| B | 能识别主要枚举、状态和异常，但部分推断需人工校验。 |
| C | 主要输出统计或字段摘要，业务推断价值有限。 |
| D | 过度推断或误读样例数据。 |

## 3.5 UI / 截图理解能力

### 测试目标

判断工具能否从业务系统截图中识别页面对象、字段、按钮、状态、动作和角色线索。

### 输入材料

- 列表页截图。
- 详情页截图。
- 编辑页截图。
- 审批页截图。
- 截图 manifest。
- 可选 OCR 文本。

### 输出要求

- `ui_ontology.json`。
- 页面到 concept/action/state 的映射。
- 字段、按钮、标签、状态和页面流程说明。
- screenshot evidence：图片路径、区域、可见文本。

### 自动评分项

- 关键字段识别率。
- 关键按钮识别率。
- 页面 source_ref 合法率。
- 编造 UI 元素数量。

### 人工评分项

- 是否正确理解页面用途。
- 是否能将 UI 元素映射到业务对象。
- 是否发现数据库/文档未体现的业务动作。
- 截图信息与其他来源融合质量。

### 红线失败项

- 工具不能看图却声称看到了具体内容。
- 编造截图中不存在的按钮、字段或状态。
- 把 OCR 错误当作确定业务事实且未标注不确定。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 能准确识别页面字段、动作和状态，并与其他来源融合。 |
| B | 能识别主要 UI 元素，复杂流程需人工校验。 |
| C | 需要人工 OCR 或文字描述辅助，能力受限。 |
| D | 无法可靠处理截图。 |

## 3.6 Web 递归分析能力

### 测试目标

判断工具能否基于离线 HTML 快照和页面地图识别系统模块、页面导航、业务流程和页面间关系。

### 输入材料

- HTML 页面快照。
- sitemap 或 page manifest。
- 页面标题和 URL 映射。
- 表单字段和菜单结构。

### 输出要求

- `web_ontology.json`。
- 页面模块清单。
- 页面到业务概念、动作和流程的映射。
- 页面间链接关系和流程路径。
- web page evidence。

### 自动评分项

- 页面覆盖率。
- 关键页面识别率。
- 链接关系识别率。
- DOM/source_ref 合法率。
- 编造页面数量。

### 人工评分项

- 业务路径理解质量。
- 导航结构与业务模块映射质量。
- 页面级发现能否补充代码/数据库证据。

### 红线失败项

- 编造未提供页面、链接或功能。
- 将静态样式或脚本错误识别为业务流程。
- 未说明页面快照不完整导致的限制。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 能跨页面递归融合，形成模块和流程视图。 |
| B | 能做页面级分析和主要链接识别。 |
| C | 只能处理单页摘要或依赖人工整理页面清单。 |
| D | 无法处理 Web 快照或大量编造。 |

## 3.7 文档理解能力

### 测试目标

判断工具能否从中文业务文档、需求文档、接口文档、Excel、PDF 转文本中抽取业务概念、规则、流程、角色和权限。

### 输入材料

- 业务说明文档。
- 需求文档。
- 接口文档。
- Excel 数据字典。
- PDF/Word 转换文本。

### 输出要求

- `doc_ontology.json`。
- 文档术语表。
- 规则清单。
- 角色和权限清单。
- 文档 evidence：文件、章节、页码、段落或引用文本。

### 自动评分项

- 关键术语召回率。
- 关键规则召回率。
- 角色识别率。
- 文档 source_ref 合法率。
- 明确规则遗漏数量。

### 人工评分项

- 中文业务语义理解。
- 规则表达是否可执行。
- 是否区分背景描述、业务约束和操作步骤。
- 是否能处理长文档分片。

### 红线失败项

- 忽略文档中明确的关键审批规则。
- 将示例说明误当作强制规则。
- 反向理解角色权限或状态流转。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 能从长文档中稳定抽取概念、规则、流程和证据。 |
| B | 能处理章节级文档，关键规则基本准确。 |
| C | 偏摘要，结构化抽取和证据定位较弱。 |
| D | 中文业务理解错误严重。 |

## 3.8 多源融合能力

### 测试目标

判断工具能否融合代码、DDL、metadata、样例数据、文档、截图和 Web 证据，去重、合并 alias、识别冲突并生成统一 ontology draft。

### 输入材料

- 单源分析输出。
- 原始多源材料。
- aliases。
- known conflicts。
- source inventory。

### 输出要求

- `merged_ontology.json`。
- `conflicts.json`。
- `evidence_map.json`。
- 同一业务对象的跨源映射。
- 冲突和不确定性说明。

### 自动评分项

- 核心概念融合召回率。
- 重复概念率。
- alias 合并准确率。
- 已知冲突召回率。
- 跨源 evidence 覆盖率。

### 人工评分项

- 抽象层次是否合适。
- 是否正确处理多源不一致。
- 是否避免简单拼接。
- 最终本体是否可用于交付讨论。

### 红线失败项

- 对明确冲突强行选择一个结论且无证据说明。
- 大量重复概念未合并。
- 无证据合并不同业务对象。
- 为了完整性编造来源。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 能证据驱动融合多源材料，主动识别冲突和不确定性。 |
| B | 能合并主要对象和关系，冲突处理需人工校验。 |
| C | 主要是多源结果拼接，重复和冲突较多。 |
| D | 融合结果互相矛盾或大量无证据。 |

## 3.9 结构化输出能力

### 测试目标

判断工具能否稳定输出可解析、字段完整、结构一致、可被后续评分和报告流程消费的 JSON。

### 输入材料

- 输出结构说明。
- ontology schema 说明。
- evidence map schema 说明。
- JSON repair prompt。

### 输出要求

- 有效 JSON。
- 不使用 markdown code fence 包裹最终机器可读文件。
- 必填字段完整。
- 数组、对象、枚举字段类型正确。
- 多轮修复后保留原始语义，不随意删减内容。

### 自动评分项

- JSON 可解析率。
- schema 合规率。
- 必填字段完整率。
- 类型错误数量。
- 修复轮次。

### 人工评分项

- 输出是否易读、可维护。
- 字段命名是否稳定。
- 是否便于后续人工复核。
- 是否为了过 schema 删除关键信息。

### 红线失败项

- 允许追问次数内仍无法输出可解析 JSON。
- 输出结构每轮大幅漂移，无法复盘。
- 为了通过格式检查删除证据或关键业务元素。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 一次或轻微修复后稳定合规。 |
| B | 两轮内可修复合规，内容基本保留。 |
| C | 需大量人工整理才能使用。 |
| D | 无法形成机器可读结构。 |

## 3.10 Agentic 交付能力

### 测试目标

判断工具能否像交付助手一样规划任务、分解步骤、读取材料、生成中间结果、自检、修复并产出最终结果。

### 输入材料

- 完整测试包。
- 任务目标。
- 操作权限说明。
- 输出结构说明。
- 时间限制和追问限制。

### 输出要求

- 执行计划。
- 分阶段中间结果。
- 自检记录。
- 修复记录。
- 最终 ontology、evidence map、conflict list。
- 明确未完成项和限制。

### 自动评分项

- 子任务完成率。
- 中间产物完整率。
- 自检问题修复率。
- 输出文件覆盖率。
- 超时或卡住次数。

### 人工评分项

- 计划是否合理。
- 是否能自主发现遗漏。
- 是否能在权限受限下选择降级路径。
- 是否减少操作员负担。

### 红线失败项

- 不读取或不引用证据，直接生成最终答案。
- 失败后不记录原因而继续输出伪完成结果。
- 将操作员临时补充的答案伪装为工具自动发现。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 能形成接近自动化流水线的端到端交付闭环。 |
| B | 能半自动分阶段执行，关键判断仍需人工。 |
| C | 主要依赖操作员逐步驱动。 |
| D | 只能做单轮问答或摘要。 |

## 3.11 证据链与可追溯能力

### 测试目标

判断每个本体元素是否能追溯到具体来源，并支持客户解释、内部复盘和后续人工审核。

### 输入材料

- ontology output。
- 原始数据源。
- source inventory。
- evidence map schema 说明。

### 输出要求

- `evidence_map.json`。
- element-to-evidence 映射。
- 每个 evidence item 至少包含 source_type、source_ref、location、summary、confidence。
- 对证据不足的元素标注 uncertainty。

### 自动评分项

- 元素 evidence 覆盖率。
- source_ref 合法率。
- 不存在引用数量。
- 证据类型覆盖数。
- 关键元素缺证数量。

### 人工评分项

- 证据是否足以支持结论。
- 证据粒度是否可复核。
- 是否区分直接证据和间接证据。
- 是否保留冲突证据。

### 红线失败项

- 引用不存在文件、表、字段、页面或文档段落。
- 为关键结论伪造证据。
- 大量本体元素无来源且未标注不确定。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 核心元素均有元素级证据，可直接复核。 |
| B | 主要对象和规则有证据，部分属性证据较粗。 |
| C | 只有文件级或对象级证据，复核成本高。 |
| D | 无证据链或引用不可信。 |

## 3.12 低经验人员可操作性

### 测试目标

判断超聚变团队中不熟悉本体方法论的交付人员是否能按手册完成评估。

### 输入材料

- Operator Playbook。
- checklist。
- 测试任务包。
- customer profile。
- prompts。

### 输出要求

- 完整 execution log。
- 原始输出保存。
- 标准化结果保存。
- 失败记录。
- 操作员反馈。

### 自动评分项

- checklist 完成率。
- 输出文件命名合规率。
- 必填记录完整率。
- 允许追问次数合规率。

### 人工评分项

- 操作员是否能独立执行。
- 卡点是否集中在工具限制还是手册不清。
- 是否需要本体专家频繁介入。
- 记录是否足以复盘。

### 红线失败项

- 低经验人员无法完成基础 Smoke Test。
- 必须由本体专家代替操作才能产出结果。
- 操作过程无法复盘。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 低经验人员可独立完成，少量问题可按手册处理。 |
| B | 需要少量远程指导。 |
| C | 需要频繁求助或专家判断。 |
| D | 不适合低经验人员执行。 |

## 3.13 人机协同成本

### 测试目标

量化完成评估任务所需的人力投入，判断 AI 工具组合是否具有交付经济性。

### 输入材料

- execution log。
- 追问记录。
- 人工修复记录。
- 专家介入记录。
- 输出质量记录。

### 输出要求

- `human_effort_metrics`。
- 每个任务记录耗时、输入次数、追问次数、重试次数、人工修复次数、专家介入次数、人工重写比例。
- 汇总 human effort index。

### 自动评分项

- 总耗时。
- 追问次数。
- 重试次数。
- JSON 修复次数。
- 输出返工次数。

### 人工评分项

- 人工负担是否可接受。
- 人工介入是流程性操作还是业务性重写。
- 工具是否实际减少交付工作量。

### 红线失败项

- 人工重写比例超过 50%。
- 专家介入成为主要产出来源。
- 工具多次卡死导致流程不可控。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 低干预，AI 产出可直接进入审核。 |
| B | 干预可接受，主要是修格式和补证据。 |
| C | 人工成本高，适合局部辅助。 |
| D | 人工重写为主，不具备交付经济性。 |

## 3.14 幻觉控制能力

### 测试目标

判断工具是否能坚持证据边界，在材料缺失、冲突或诱导场景下避免编造，并主动表达不确定性。

### 输入材料

- 缺字段材料。
- 缺文档材料。
- 冲突材料。
- 陷阱字段。
- 幻觉红线测试 prompt。

### 输出要求

- 明确区分事实、推断和未知。
- 对无证据内容标注 uncertainty。
- 对冲突内容输出 conflict。
- 不编造来源。

### 自动评分项

- 严重幻觉数量。
- 普通幻觉数量。
- 不存在 source_ref 数量。
- 无证据规则数量。
- uncertainty 覆盖率。

### 人工评分项

- 风险判断是否保守。
- 是否敢于回答“无法判断”。
- 是否把合理推断表达为推断而非事实。

### 红线失败项

- 编造关键表、字段、代码、规则或来源。
- 对明确缺失信息给出确定答案。
- 对冲突材料强行生成单一结论且无说明。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 主动标注不确定和冲突，几乎无幻觉。 |
| B | 少量可纠正幻觉，不影响核心判断。 |
| C | 常见无证据推断，需要人工严审。 |
| D | 幻觉不可控，不适合本体项目。 |

## 3.15 中文政企业务理解能力

### 测试目标

判断工具是否能理解中文政企场景中的业务术语、审批流程、角色权限、组织结构和常见管理规则。

### 输入材料

- 中文业务说明。
- 中文字段注释。
- 中文审批规则。
- 中文 UI 文案或页面字段。
- 行业术语表。

### 输出要求

- 中文业务概念清单。
- 角色和权限清单。
- 状态流转和审批规则。
- 行业术语 aliases。
- 中文 evidence 引用。

### 自动评分项

- 关键中文术语召回率。
- 角色识别率。
- 权限识别率。
- 状态流转识别率。
- 术语 alias 匹配率。

### 人工评分项

- 中文语义是否准确。
- 政企业务流程是否合理。
- 是否误解常见审批、归档、验收、付款等术语。
- 是否能处理中文缩写、旧系统字段和业务别名。

### 红线失败项

- 把关键中文业务概念完全误读。
- 把审批角色、申请角色或财务角色混淆并影响核心流程。
- 将中文字段按字面错误翻译为无关业务含义。

### 等级解释

| 等级 | 解释 |
|---|---|
| A | 中文政企业务语义准确，可支撑客户讨论。 |
| B | 通用中文业务理解可用，行业细节需人工校验。 |
| C | 需要业务专家频繁解释术语。 |
| D | 不适合中文政企业务本体项目。 |
