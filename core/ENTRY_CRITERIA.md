# Entry Criteria

## 1. 目标

本文件定义客户现场 AI 工具组合进入真实本体项目 POC 前的内部准入标准。准入结论必须基于完整工具组合，而不是单独模型或单独工具品牌。

被判断对象：

```text
客户环境 + AI 工具 + 底层模型 + 权限边界 + 可用数据源 + 操作人员 + 标准任务包
```

## 2. 准入结论类型

| 结论 | 含义 | 推荐策略 |
|---|---|---|
| Approved as Primary | 可作为主力工具进入 POC。 | 可承担本体草案生成主流程，FDE 专家审核。 |
| Approved with Guardrails | 有条件进入 POC。 | 限定范围，关键环节由 FDE 核心团队看护。 |
| Auxiliary Only | 只能作为辅助工具。 | 用于单源分析、文档摘要、字段解释、初步草案。 |
| Local Use Only | 只能局部使用。 | 只用于低风险、低复杂度任务。 |
| Not Recommended | 不建议用于本体项目。 | 要求客户补充工具、模型、权限或调整 POC 范围。 |
| Blocked / Inconclusive | 无法判断。 | 先补充权限、数据导出或评估条件后重测。 |

## 3. 最低准入门槛

进入真实客户 POC 前，建议同时满足：

- Smoke Test 不低于 B-。
- Standard Test 综合不低于 B-。
- Critical 红线数量为 0。
- 数据库 Metadata 理解能力不低于 B。
- 代码理解能力不低于 B-。
- 文档理解能力不低于 B-。
- 多源融合能力不低于 B-。
- 结构化输出能力不低于 B。
- 证据链与可追溯能力不低于 B-。
- 幻觉控制能力不低于 B。
- 低经验人员可操作性不低于 B-。
- Human Effort Index 不高于 2。
- 能保存完整原始输出、操作日志和评分结果。

## 4. 主力工具准入标准

若要作为 POC 主力工具，建议满足：

- Standard Test 综合不低于 B+。
- 数据库 Metadata 理解能力不低于 B+。
- 代码理解能力不低于 B。
- 文档理解能力不低于 B。
- 多源融合能力不低于 B。
- 结构化输出能力不低于 B+。
- 证据链与可追溯能力不低于 B。
- 幻觉控制能力不低于 B+。
- 低经验人员可操作性不低于 B。
- Human Effort Index 不高于 2。
- 无 Critical 红线。
- Major 红线可控且有明确 guardrails。

推荐使用方式：

- 本体草案初稿生成。
- 多源材料初步融合。
- 规则和证据初步整理。
- FDE 专家审核前的结构化预处理。

## 5. 有条件准入标准

若工具组合达到 B- 到 B 区间，可有条件进入 POC。

适用条件：

- Critical 红线为 0。
- 结构化输出能力至少 B-。
- 数据库或文档至少一个核心输入能力达到 B。
- Human Effort Index 不高于 3。
- 客户接受缩小 POC 范围。
- FDE 核心团队可参与关键审核。

限制方式：

- 只做单源或双源本体草案。
- 不自动处理关键冲突。
- 不让工具直接生成对客户承诺的最终本体。
- 所有关键规则、权限、金额阈值必须人工复核。

## 6. 只能辅助使用的标准

出现以下情况，工具只能作为辅助：

- Standard Test 综合 C+ 到 B-。
- 多源融合能力低于 B-。
- 证据链能力低于 B-。
- 结构化输出需要频繁人工修复。
- 人机协同成本 HEI = 3。
- 工具只能处理文档或单文件，不能处理多源材料。

推荐用途：

- 文档摘要。
- 字段说明翻译。
- 单表解释。
- 简单概念候选生成。
- 操作员辅助问答。

不推荐用途：

- 多源融合。
- 冲突处理。
- 关键规则抽取。
- 自动生成客户可评审本体草案。

## 7. 不建议使用的标准

出现以下任一情况，不建议用于本体项目：

- Critical 红线 >= 1。
- Smoke Test 低于 C+。
- Standard Test 综合低于 C。
- 结构化输出能力 D。
- 幻觉控制能力 D。
- 证据链能力 D。
- Human Effort Index >= 4。
- 低经验操作员无法完成 Smoke Test。
- 客户权限限制导致核心数据源全部 Blocked。

处理建议：

- 要求客户补充更强工具或模型。
- 要求客户提供文件读取、metadata 导出或结果保存能力。
- 调整 POC 范围为人工主导、AI 辅助。
- 暂缓本体自动化 POC。

## 8. Blocked / Inconclusive 标准

如果关键能力无法测试，不得给出过度乐观结论。

标记 Blocked / Inconclusive 的情况：

- 无法访问代码，也无法提供代码片段。
- 无法提供 DDL、metadata 或数据字典。
- 无法保存模型输出。
- 无法确认模型和工具版本。
- 客户不允许记录操作日志。
- 工具权限与真实 POC 权限明显不同。

处理建议：

- 补充权限后重测。
- 使用脱敏导出材料重测。
- 明确报告“无法判断真实 POC 能力”。
- 不将结果写入正式 benchmark，或仅作为受限画像。

## 9. 不同项目策略

### 9.1 可作为主力工具

条件：

- 满足主力工具准入标准。
- 无 Critical 红线。
- HEI <= 2。

策略：

- AI 工具生成初稿。
- FDE 专家审核和修订。
- 客户业务专家确认。

### 9.2 可作为辅助工具

条件：

- 达到有条件准入或辅助使用标准。
- 单源能力较强，但多源融合或证据链不足。

策略：

- 用于代码、文档、DDL 的局部分析。
- 多源合并由 FDE 主导。
- 工具输出不能直接作为最终草案。

### 9.3 只能局部使用

条件：

- 仅部分能力达到 B。
- 其他关键能力 C 或 Blocked。

策略：

- 限定任务边界。
- 明确输出不进入最终交付，除非人工确认。

### 9.4 不建议用于本体项目

条件：

- Critical 红线。
- 结构化和证据能力不可用。
- 幻觉不可控。

策略：

- 更换工具组合。
- 补充权限后重新评估。
- POC 改为人工主导。

## 10. 需要 FDE 核心团队介入的条件

出现以下情况，需要 FDE 核心团队介入：

- Standard Test 仅 B- 或 C+，但业务上仍希望进入 POC。
- 多源融合能力低于 B。
- 冲突处理能力不足。
- 证据链能力低于 B-。
- 客户工具不能稳定输出 JSON。
- HEI = 3。
- Major 红线存在但可缓解。
- 行业语义复杂，低经验人员无法判断。

介入方式：

- 设计客户专用降级流程。
- 复核关键本体元素。
- 复核规则、权限、状态流转。
- 判断是否缩小 POC 范围。
- 确认最终报告准入结论。

## 11. 需要客户补充能力或权限的条件

应要求客户补充：

- 代码仓库只读访问或代码导出。
- DDL、metadata、数据字典导出。
- 脱敏样例数据。
- 文档批量导出能力。
- 结果保存和日志留存能力。
- 工具稳定 JSON 输出能力。
- 截图或页面快照导出。
- 数据库只读连接或 metadata 导出。

## 12. 准入决策记录格式

每次评估报告必须记录：

```text
entry_decision:

  rationale:
  required_conditions:
  prohibited_usage:
  recommended_usage:
  fde_involvement_required:
  customer_actions_required:
  redline_summary:
  score_caps_applied:
  next_review_date:
```

## 13. 示例决策口径

### 示例 1：Approved with Guardrails

工具组合综合 B，数据库和文档能力较好，但多源融合和证据粒度不足。

建议：

- 可进入限定范围 POC。
- 只允许用于 DDL、metadata、文档的单源抽取和初步融合。
- 冲突处理和最终本体评审由 FDE 核心团队负责。

### 示例 2：Auxiliary Only

工具组合文档理解 B+，但代码读取和 JSON 输出能力 C。

建议：

- 只能用于文档摘要、术语抽取和规则候选。
- 不用于自动生成标准 ontology JSON。
- 需要人工整理输出。

### 示例 3：Not Recommended

工具组合出现编造表字段和伪造 source_ref。

建议：

- 不进入真实 POC 主流程。
- 要求更换模型或补充更可靠工具。
- 如客户坚持使用，只能作为非交付性探索工具。
